# PDF Export Charts Not Rendering — Postmortem

**Date**: 2026-05-07  
**Issue**: Annual/Monthly PDF exports returned text-only fallback pages (4KB) instead of full reports with charts (expected ~200KB+)  
**Severity**: Critical — user-facing feature completely broken

## Root Cause

The deployed `dist/` directory at `/var/www/finance-manager.clodhost.com/frontend/dist/` was **missing three files**:

- `export.html` — Annual Financial Report template (Chart.js-based)
- `export-monthly.html` — Monthly Financial Report template
- `chart.umd.min.js` — Chart.js UMD bundle (needed by the templates)

### How it manifested

1. The Express backend's static middleware serves files from `frontend/dist/` (see `backend/index.js` line ~481)
2. When Puppeteer navigated to `http://localhost:3847/export.html`, Express' `express.static` didn't find the file in `dist/`
3. The request fell through to the SPA catch-all route `app.get('*', ...)` which returned `frontend/index.html` (a 726-byte SPA shell)
4. Chart.js never loaded because `chart.umd.min.js` wasn't served either
5. The backend's Puppeteer catch block caught the error and returned a text-only fallback: "Charts could not be rendered"

### Why `dist/` was missing these files

The **deploy script** (`deploy.sh`) copied `dist/*` to the live **frontend root** (`/var/www/.../frontend/`), which is where Apache serves from. However, the **backend's Express static middleware** expects files in `frontend/dist/`. The deploy script did NOT copy to the `dist/` subdirectory.

The files existed in the source repo's `frontend/` root (Vite copies them into `dist/` during build via a custom `copy-export-html` plugin), but the deploy script only targeted the Apache-served location, not the backend's static directory.

## Why It Took Multiple Attempts to Fix

Several factors compounded to make this hard to diagnose:

1. **Two separate directory trees**: Source at `/tmp/finance-manager/` vs deployed at `/var/www/finance-manager.clodhost.com/`. The source had correct files; the deployed version didn't.
2. **No error signals in the API response**: The backend returns HTTP 200 with a fallback HTML message instead of an error — there's no way to tell from the API response alone whether charts rendered.
3. **Indirection in the serving architecture**: Apache serves frontend files directly → Express middleware serves from `dist/` subdirectory → SPA catch-all hides 404s → Puppeteer silently falls back. Each layer obscured the root cause.
4. **The deploy script was silently incomplete**: It copied `dist/*` to one location but not the other. No error was thrown because the copy succeeded — it just copied to the wrong target for the backend's needs.
5. **Previous debugging attempts focused on code logic**: We checked the Puppeteer pipeline, Chart.js loading, the export page JS — all of which were correct in the source. The bug was purely in file placement on the production server.

## What Finally Fixed It

### The key insight: test from the OUTSIDE

Instead of reading code and reasoning about what should happen, we checked what was **actually being served**:

```bash
curl -s http://localhost:3847/export.html | wc -c
# → 726 bytes (clearly not the 15KB export template — it's the SPA shell!)
```

This immediately revealed the file wasn't being served. From there, checking the deployed `dist/` directory confirmed the files were missing.

### The e2e test that validates the fix

The critical prompt that drove the fix was: **"please check by debugging and running a test for puppeteer export on backend, we should use backend to retrieve the PDF and a test should confirm that the HTML does indeed contain and render charts"**

This led to creating `test-pdf-export.js`, which:

1. Launches Puppeteer in headless mode
2. Injects sample data via `evaluateOnNewDocument` (matching backend behavior)
3. Navigates to the export HTML pages
4. Verifies Chart.js loaded
5. Verifies data was injected
6. Waits for `window.__RENDER_DONE__` signal
7. Samples canvas pixel data to confirm charts actually drew content
8. Generates PDF and verifies it's >10KB (charts present, not text-only)
9. Also calls the live API endpoints to verify end-to-end

**The test provides concrete, falsifiable evidence**: it checks canvas pixel counts, PDF sizes, and reports exactly which chart rendered and which didn't.

## Fixes Applied

1. **Deploy script** (`deploy.sh`): Added second copy operation to also place `dist/*` into `frontend/dist/`:
   ```bash
   mkdir -p "$LIVE_FRONTEND/dist"
   cp -r "$FRONTEND_DIR/dist"/* "$LIVE_FRONTEND/dist/"
   ```

2. **PDF templates**: Removed "Generated by Finance Manager" footer text, beautified date format from `5/7/2026` to `May 7, 2026`, cleaned up header subtitle.

## Preventive Measures

- The `test-pdf-export.js` script is committed to the repo and should be run after each deployment to verify PDF export works end-to-end
- The script tests both sample data rendering AND live API endpoints
